Docs

A complete walkthrough of the framework: how to write a world, how to compose a scenario out of users and agents, how verification works, and how to drive the resulting environment as part of an RL training loop.

What Ensemble is

Ensemble is a framework for building RL environments in which several simulated users and several agents share one typed, mutable world. You describe the world in Rust because it owns mutable state and needs to be fast. You write the scenarios in Python because that is where you want iteration to be cheap. The framework handles message routing, tool dispatch, the trace log, and the grader that turns a finished rollout into a numerical reward.

Most agent benchmarks model one agent talking to one human in a stateless sandbox. Production systems do not look like that. Real support, sales, customer success, and operations involve multiple users contending for attention, multiple agents seeing each other's tool calls, shared state that changes underfoot, and policies that only become visible across many turns. Ensemble lets you build that, run rollouts at scale, and turn the rollouts into training signal.

The mental model

A run of Ensemble has three pieces. The first is the world, which is the typed mutable state of the simulation. The world implements a small Rust trait called WorldState. Plank's world is a SQLite database with tables for users, tickets, subscriptions, knowledge-base articles, and an audit log. Your world might be a fake CRM, a make-believe Linux box, or an in-memory graph; the framework does not care, as long as you can apply a tool call and serialize the resulting diff.

The second piece is the set of actors. Each actor is a Tokio task with its own inbox. A user actor wraps a language model and represents a simulated end user. An agent actor wraps the same kind of model and also holds a tool registry, so it can issue tool calls that mutate the world. Actors talk to each other on a shared bus, and every message is appended to an event log with a tick number and a timestamp.

The third piece is the scenario. A scenario decides which world to instantiate, spawns the actors, seeds any initial messages or tool calls, picks an until condition that decides when the simulation stops, and then defines the grader that turns the finished trace into a reward. Most scenarios are eight or ten lines of Python.

A finished run produces a trace, which is a JSONL stream of typed events that you can replay in the browser, ship to a training pipeline, or assert against in tests. The framework treats the trace as the canonical output, not the model's text. Anything you want to grade has to leave a footprint in the trace, which is why every tool call and every state change is logged automatically.

Install

You need Rust 1.80 or later, Python 3.10 or later, and uv. Clone the repo and run: 

uv sync                       # build the ensemble + plank extensions, install the workspace
cargo build -p ensemble-cli   # produces ./target/debug/ensemble
cargo test --workspace        # rust tests
uv run pytest tests/          # python tests

uv sync uses maturin to build two extensions: the ensemble._native module that backs the Python package, and a separate plank._native for the example world. Plank is a plugin: nothing in the core framework references it. Register it on first install so the scenarios know how to find it: 

./target/debug/ensemble worlds add plank examples/plank

Torch and the rest of the training stack are gated behind an extra so the base install stays small: 

uv pip install 'ensemble-train[torch]'

To run scenarios against real models, set the API key for whichever backend you want to use. Ensemble's backends auto-select from the environment when you pass backend="auto": 

export ANTHROPIC_API_KEY=sk-ant-...
# or
export OPENAI_API_KEY=sk-...

Without any keys, the default backend is the deterministic mock, which is what the test suite uses and what bakes the trace on the front page.

Your first scenario

Reference: scenarios.

A scenario is a Python coroutine registered with the @scenario decorator. The smallest interesting one looks like this: 

# scenarios/hello.py
from collections.abc import AsyncIterator

from ensemble import Until, World, scenario


@scenario("hello", world="noop")
async def hello(world: World) -> AsyncIterator[Until | dict[str, float]]:
    alice = world.spawn_user(id="alice", model="claude-haiku-4-5")
    rep = world.spawn_agent(id="rep", model="claude-sonnet-4-5")
    alice.say("rep", "hi")

    yield world.until(world.turn_count > 4)
    yield {"saw_reply": 1.0}

The world="noop" kwarg names which world this scenario expects. The runner uses it as the default when --world isn't supplied on the CLI; passing an explicit world to the runner overrides the decorator's choice, which is useful for cross-world testing.

Three things happen here. The scenario sets up the actors and seeds a message. It yields an until condition, which tells the scheduler to start the simulation and run it until that condition fires. Then it yields a dictionary of named scores, which becomes the grader output.

Run the scenario from Python with the synchronous helper: 

from ensemble import RunResult, run_scenario

result: RunResult = run_scenario(
    "hello",
    world_name="noop",
    backend="anthropic",
)
print(result.scores)
print(len(result.trace), "events")

The noop world is a built-in placeholder with no state and no tools, useful for tests. Real scenarios target a real world by name, such as plank. The backend argument chooses the LLM client: "mock" for offline determinism (the default), "anthropic" or "openai" for the cloud providers, "vllm" for a self-hosted endpoint, or "auto" to pick the first one whose API key is in the environment.

Same thing from the command line:

./target/debug/ensemble run hello --world noop
# writes ./traces/hello.jsonl

Your first world

Reference: world api.

Worlds are plugins. Each world is a directory with three things: a rust crate that owns the state and tool implementations, a python package that ensemble imports to register the world, and a world.toml manifest that ties them together and declares default personas and resources. The framework discovers a world by name through ~/.ensemble/worlds.toml; that registry is managed with ensemble worlds add <name> <path>.

Run ensemble init my_world to get a working skeleton, then open my_world/world/src/lib.rs. A world implements the WorldState trait: 

pub trait WorldState: Send + Sync + 'static {
    type ToolCall:   DeserializeOwned + Send;
    type ToolEffect: Serialize + Send;
    type Diff:       Serialize + Send;

    fn apply(
        &mut self,
        call: Self::ToolCall,
    ) -> Result<(Self::ToolEffect, Self::Diff), ToolError>;

    fn snapshot(&self) -> Vec;
    fn restore(&mut self, snapshot: &[u8]) -> Result<(), RestoreError>;
}

The three associated types let you express what an agent can ask of the world, what it gets back, and what changed. The Diff is the most important of the three. It shows up in the trace as a state_diff event, and the trace viewer renders it on the right-hand side. A good diff identifies the table that changed, the row and field, and the old and new values, so a reader of the JSONL can reconstruct the run without rerunning it.

Here is a tiny but complete world that holds a counter:

use ensemble_core::error::{RestoreError, ToolError};
use ensemble_core::world::WorldState;
use serde::{Deserialize, Serialize};

#[derive(Default)]
pub struct Counter {
    value: i64,
}

#[derive(Deserialize)]
#[serde(tag = "op", rename_all = "snake_case")]
pub enum Call {
    Inc { by: i64 },
    Reset,
}

#[derive(Serialize, Clone)]
pub struct Effect {
    new_value: i64,
}

#[derive(Serialize)]
pub struct Diff {
    field: &'static str,
    old: i64,
    new: i64,
}

impl WorldState for Counter {
    type ToolCall = Call;
    type ToolEffect = Effect;
    type Diff = Diff;

    fn apply(&mut self, call: Call) -> Result<(Effect, Diff), ToolError> {
        let old = self.value;
        match call {
            Call::Inc { by } => self.value += by,
            Call::Reset => self.value = 0,
        }
        Ok((
            Effect { new_value: self.value },
            Diff { field: "value", old, new: self.value },
        ))
    }

    fn snapshot(&self) -> Vec {
        serde_json::to_vec(&self.value).unwrap()
    }

    fn restore(&mut self, snap: &[u8]) -> Result<(), RestoreError> {
        self.value = serde_json::from_slice(snap)
            .map_err(|e| RestoreError::Decode(e.to_string()))?;
        Ok(())
    }
}

snapshot and restore exist so the runtime can checkpoint a world before a risky branch and roll back if needed. They are also what let you run counterfactual rollouts from the same starting state during training.

The world's python package wires the rust crate into the plugin registry. A minimal one looks like: 

# my_world/my_world/__init__.py
from pathlib import Path
from ensemble import PluginTool, register_world
from . import _native   # built by maturin alongside ensemble._native

def _setup():
    db = _native.MyState()  # fresh per World instance
    tools = [
        PluginTool(
            name="inc",
            description="Increment the counter by `by`.",
            parameters={
                "type": "object",
                "properties": {"by": {"type": "integer"}},
                "required": ["by"],
            },
            fn=lambda args_json: db.dispatch("inc", args_json),
        ),
    ]
    return tools, []

register_world(
    "my_world",
    setup=_setup,
    personas_dir=Path(__file__).parent.parent / "personas",
)

Tools and how agents call them

Reference: tools.

Agents do not poke at the world directly. They issue tool calls, and the tool registry dispatches each call to a closure that mutates the world and returns a JSON effect. The agent sees the effect on its next turn, and the rest of the system sees a pair of tool_call and tool_result events in the trace.

A tool is a name, a description, a JSON Schema, and a closure:

use std::sync::Arc;

use ensemble_core::error::ToolError;
use ensemble_runtime::{Tool, ToolRegistry};
use parking_lot::Mutex;
use serde_json::json;

let mut tools = ToolRegistry::new();
let state: Arc> = state.clone();

tools.register(Tool::new(
    "lookup_user",
    "Look up a user by id.",
    json!({
        "type": "object",
        "properties": { "user_id": { "type": "string" } },
        "required": ["user_id"]
    }),
    move |args| {
        let user_id = args["user_id"]
            .as_str()
            .ok_or_else(|| ToolError::InvalidArgs("user_id".into()))?;
        let user = state.lock().lookup_user(user_id)?;
        Ok(json!({ "ok": true, "data": user }))
    },
));

The description and the schema are both shipped to the language model whenever it has to decide what to call. The closure runs synchronously on the actor's thread. If you need to do anything that could block, do it inside the lock and keep the critical section short.

When the agent's model returns a tool call, the agent actor dispatches it through this registry, writes the result to the trace, and feeds the result back into the model on the next turn as a synthetic message. You do not write any of that glue yourself.

Progress, timeouts, resources, costs

Tools have four optional capabilities the runtime applies without changing the closure's signature.

A long-running tool can report progress through a ProgressEmitter parameter; the runtime buffers the entries and flushes them to the trace as progress events right before the trailing tool_result. Tool::with_timeout(Duration) caps a single dispatch; when it fires the trace gains a tool_timeout event and the calling agent sees a tool error (the scenario continues). Tool::with_resources(["billing_db"]) declares named locks the runtime acquires before invoking the tool; concurrent dispatches sharing a name serialize through the world's ResourceManager.

A tool can also annotate its outcome with costs in any open-string unit ("usd", "tokens_in", "gpu_seconds"). After each dispatch the runtime bumps the world's running total and emits a cost event. Scenarios cap spend with world.set_budget(unit, amount); when a recorded cost would push the total past the cap the scheduler halts with StopReason::BudgetExceeded and the trailing system note records why.

Emitting progress from python tools

A python tool that declares an emit_progress parameter receives a callable: each invocation records one entry that the runtime flushes to the trace as a progress event right before the trailing tool_result. The helper inspects the function's signature, so a tool that does not need progress reporting simply omits the parameter and the call shape stays the same. 

from ensemble import tool


def reconcile(user_id: str, months: int, emit_progress):
    for i in range(1, months + 1):
        emit_progress(i / months, f"scanned month {i}/{months}")
        ...  # do the actual reconciliation
    return {"ok": True, "months_reconciled": months}


reconcile_tool = tool(
    name="reconcile_billing",
    description="Walk a user's billing history and emit progress.",
    parameters={
        "type": "object",
        "properties": {
            "user_id": {"type": "string"},
            "months": {"type": "integer"},
        },
        "required": ["user_id", "months"],
    },
    fn=reconcile,
)

The runtime buffers the entries until the tool returns, then flushes them in order; the trace viewer renders each progress event with its fraction and message. Pair this with Tool::with_timeout(...) to make long-running work observable while still bounded.

Sandboxed tools

A tool that runs untrusted or fragile code (a generated GPU kernel, a binary that occasionally segfaults, an inference call that leaves the CUDA context in a bad state) should not run in the scheduler's process, because a crash takes the whole run with it. Plugin tools declared with PluginTool(..., sandbox=True) get a subprocess wrapper: each dispatch runs in a fresh python -m ensemble.tool_worker process that re-imports the world's python package, looks up the tool by name, runs it once with the supplied JSON args, prints the result envelope on stdout, and exits. 

PluginTool(
    name="run_kernel",
    description="Compile and time a CUDA kernel.",
    parameters=KERNEL_SCHEMA,
    fn=run_kernel_inline,
    sandbox=True,
)

The subprocess starts clean: any state the parent's closures captured is not shared with the worker. Tools that fit this path encode their entire input in the JSON args and produce their entire output in the JSON return; anything that needs to consult parent-process state stays in-process. A crash in the worker surfaces as a structured error envelope, not as a scheduler exit; the calling agent sees a tool error and the scenario continues. The cost is process startup per dispatch (tens to hundreds of milliseconds), so reach for sandbox=True when the safety property is worth more than the latency.

Budgets, world-wide and per-actor

Two cost ledgers run side by side. The world-wide ledger tracks every cost recorded under a given unit; the per-actor ledger tracks each actor's own running total. A cost recorded against an actor counts against both. Either ledger can carry a cap, and either cap firing halts the scheduler. 

world.set_budget("usd", 0.50)                     # world-wide
world.set_budget("usd", 0.10, actor="alice")      # alice may spend at most $0.10
world.set_budget("tokens_out", 50_000)            # cap on output tokens across the run

world.cost_total("usd")                           # world-wide running total
world.cost_total("usd", actor="alice")            # alice's own running total

Use the per-actor variant when one slot's budget should not affect another's. A scenario that wants to make sure the trained-persona user never spends more than $0.05 per run while leaving the agent free to use its full budget caps alice and leaves the world-wide ledger alone: 

world.set_budget("usd", 0.05, actor="alice")

The cap is checked after each record_cost, so a single tool dispatch that pushes the running total past the cap halts as soon as the dispatch returns. The trailing system event names the unit, the offending amount, and the cap, so the trace is enough to reconstruct what went over. Tools annotate cost by returning a costs dict from their plugin envelope; LLM backends record tokens_in, tokens_out, and (when the model is in crates/ensemble-runtime/pricing.toml) usd automatically against the actor that issued the call.

Users and personas

Reference: personas.

A user actor is a wrapper around an LLMBackend, but in practice you almost always want it to behave like a particular person. That is what personas are for. A persona is a TOML file with a style description, a hidden-state schema, a system prompt template, and optionally a set of training hyperparameters. 

[persona]
name = "frustrated_power_user"
mode = "trained"           # or "prompted"
description = "Long-tenured paid user with low patience."

[persona.style]
tone = "frustrated"
verbosity = "high"

[persona.hidden_state.schema]
mood = { type = "string", default = "annoyed" }
hidden_goal = { type = "string", default = "refund_3mo" }

[persona.system_prompt]
template = """
You are a long-tenured customer. You want a refund for the last three months.
Do not mention that hidden goal until at least the third turn.
"""

[persona.training]
base_model = "Qwen/Qwen2.5-7B-Instruct"
backend = "modal"
adapter_name = "ensemble-plank-frustrated-power-user"

Prompted personas live entirely in their system prompt. The framework renders the hidden state into a tagged block, instructs the model not to reveal it, and otherwise gets out of the way. Trained personas swap the frontier model for a smaller fine-tuned one, trained with DPO on a preference dataset that rewards in-character continuations. The scenario does not care which kind it got; both implement the same LLMBackend trait.

Hidden state is the lever that lets a user actor stay coherent across a long run. The hidden goal travels with the actor, and when the scenario's grader needs to check whether Alice actually got what she wanted, it reads alice.hidden_state rather than trying to parse the transcript.

Verification and reward

Verification in Ensemble is the second yield. After the simulation stops, the scenario produces a dictionary of named scalar scores. Each score is a number, conventionally between zero and one, that captures one named outcome. A simple scenario might have one score. The refund_storm scenario produces four, one per user plus a global policy check. 

from collections.abc import AsyncIterator

from ensemble import Until, World, scenario


@scenario("refund_storm", world="plank")
async def refund_storm(world: World) -> AsyncIterator[Until | dict[str, float]]:
    alice = world.spawn_user(
        id="alice",
        persona="frustrated_power_user",
        hidden_goal="refund_3mo",
        model="claude-haiku-4-5",
    )
    bob = world.spawn_user(
        id="bob",
        persona="confused_new_user",
        model="claude-haiku-4-5",
    )
    rep1 = world.spawn_agent(
        id="rep1",
        model="claude-sonnet-4-5",
        tools=["lookup_user", "issue_refund", "escalate"],
    )

    alice.act("open_ticket", subject="want my money back")
    bob.act("open_ticket", subject="how do refunds work")

    yield world.until(world.turn_count > 30)

    yield {
        "alice_refund_resolved": float(alice.hidden_goal_resolved()),
        "bob_no_upsell": float(not bob.was_upsold()),
        "no_double_refunds": float(not world.had_double_refund()),
    }

The grader has three places to look. Hidden actor state, which the persona maintains and updates as the conversation progresses. The world itself, which exposes world-specific predicates like had_double_refund that query the underlying SQLite tables. And the trace, which the grader can walk directly when it needs to count tool calls or detect a sequence of events.

You can split the work however you like. Cheap checks belong in the grader. Expensive ones, including LLM judges, usually live in a separate evaluation pass that consumes the saved trace later. Both produce the same kind of output: a dictionary of named scalars that training code knows how to consume.

Using Ensemble as an RL environment

Every scenario is an RL environment. A rollout is one execution of the scenario from setup through the second yield. The agent acts, the world responds, the rollout terminates when the until condition fires, and the grader produces a reward dictionary. Training a policy against the environment is then a question of generating many rollouts, computing reward, and feeding the (trace, reward) pairs into whatever optimizer you favor.

The framework makes a few design choices that matter for training.

  • Reward is multi-dimensional by default. Returning a single number throws away signal. If the grader produces five named scores, you can keep them separate, weight them differently per actor, or aggregate them at training time. None of that information is forced together at scenario authoring time.
  • State is checkpointable. Because every world implements snapshot and restore, you can branch a rollout, try several agent responses from the same state, and use the divergence in reward as a credit-assignment signal. This is what makes algorithms like DPO and rejection sampling practical.
  • The trace is the policy's experience. Anything the agent observes in a real run also shows up in the trace. The trace is therefore directly usable as the experience tape for offline RL, and as the input to a reward model trained on graded rollouts.
  • The user side is also learnable. User actors are backed by the same LLMBackend trait as agents, so a trained persona is itself a learned policy. The ensemble-train package ships a DPO+LoRA pipeline that teaches a persona to stay in character against an adversarial breaker.

The minimal training loop looks like this:

from ensemble import RunResult, run_scenario


def rollout() -> RunResult:
    return run_scenario(
        "plank.refund_storm",
        world_name="plank",
        backend="anthropic",
    )


batch: list[RunResult] = [rollout() for _ in range(64)]

for r in batch:
    alice = r.scores["alice_refund_resolved"]
    policy_violation = r.scores.get("policy_violation", 0.0)
    reward = alice - policy_violation
    optimizer.step(trace=r.trace, reward=reward)

Replace the optimizer call with TRL, verl, or whatever you are using. The shape stays the same.

Mid-run intervention

Reference: scenarios.

The yield flavor covers most cases, but sometimes you want to inspect or modify the simulation while it is running. For that, scenarios can use an async context manager: 

from ensemble import World, scenario


@scenario("enterprise_audit", world="plank")
async def enterprise_audit(world: World) -> dict[str, float]:
    carol = world.spawn_user(
        id="carol",
        persona="enterprise_admin",
        hidden_goal="audit_log_export",
        model="claude-haiku-4-5",
    )
    rep = world.spawn_agent(
        id="rep1",
        model="claude-sonnet-4-5",
        tools=["escalate", "search_kb"],
    )
    carol.say("rep1", "i need the q3 audit log exported as csv.")

    async with world.simulate() as run:
        await run.wait_until(world.turn_count > 10)
        carol.say("rep1", "if you cannot do this, please escalate.")
        await run.wait_until(world.turn_count > 18)

    return {"escalated_or_promised": 1.0}

Inside the async with block the scheduler runs on a background task. wait_until blocks the scenario coroutine until the condition fires or a timeout elapses. Between the waits you can inspect hidden state, push a new message into an actor's inbox, mutate the world directly, or assert intermediate invariants. When the block exits, the scheduler is stopped and the trace is final.

The async-with surface

world.simulate() returns an async context manager. Entering it starts the scheduler on a background tokio task and hands you a SimulationRun with a single method: await run.wait_until(condition, timeout_ms=30_000). The condition is either an Until (typically world.turn_count > N) or anything world.until(...) accepts. The call returns True when the predicate fires and False when the timeout elapses; the scheduler keeps running either way, so a returning-false case can be followed by another wait against a different condition. Between waits the python coroutine and the rust scheduler share the world: a user.say(...) issued from python lands in the next delivery cycle, and a world.evaluate_predicate(...) sees whatever events have already been appended.

Choose this path when the scenario needs to react to what happened earlier in the run: time pressure, dynamic injection of new prompts, a mid-run grader that decides whether to continue, or anything else the yield form cannot express because it commits to a single until-predicate up front. For scenarios that fit "seed actors, run until N turns, score the trace", stick with the yield flavor; it has fewer moving parts.

Declarative TOML scenarios

Reference: scenarios.

When you want to enumerate a matrix of similar scenarios without writing one Python function per scenario, you can declare them in scenarios.toml. The TOML compiles to the same runtime as the Python decorator: 

[scenario.refund_storm]
world = "plank"
duration_turns = 30

[[scenario.refund_storm.users]]
id = "alice"
persona = "frustrated_power_user"
model = "claude-haiku-4-5"
initial_action = { tool = "open_ticket", args = { subject = "want my money back" } }

[[scenario.refund_storm.agents]]
id = "rep1"
model = "claude-sonnet-4-5"
tools = ["lookup_user", "issue_refund"]

[scenario.refund_storm.graders]
alice_resolved = "hidden_goal_resolved"
no_double_refund = "not had_double_refund"

Grader expressions use a small safe boolean DSL. The parser recognizes and, or, not, parentheses, and named predicates that the world exposes. Names that look like function calls are rejected by the parser, and there is no eval or attribute access anywhere in the pipeline. If you need a richer grader, write the scenario in Python.

The ensemble CLI

Reference: cli.

The ensemble binary is a thin wrapper around the rest of the framework. The subcommands cover the common workflows.

commandwhat it does
ensemble worlds add <name> <path> Registers a world (a directory with a world.toml manifest) under a short name. Stored at ~/.ensemble/worlds.toml. Pair with worlds list, worlds show <name>, worlds remove <name>.
ensemble init <name> Scaffolds a new world project: rust crate skeleton, python package skeleton, a world.toml, a sample scenario, and a placeholder persona. Pass --world <existing> to scaffold just a scenarios directory bound to an already-registered world.
ensemble run <scenario> --world <name> Resolves <name> through the worlds registry, imports the world's python package (which fires its register_world call), runs the scenario, and writes the trace to ./traces/.
ensemble mcp serve --world <name> --scenario <id> --as-agent <slot> Runs the named scenario with the named agent slot driven by an external MCP-aware client (Claude Code, Claude Desktop, etc.). See the MCP page.
ensemble trace view <path> Serves the static trace viewer on a local port with the given trace baked in.
ensemble train <persona.toml> Hands the persona spec off to the Python training pipeline. Supports --backend modal, --backend skypilot, and --backend local.

Training personas

Reference: personas, runtime.

Prompted personas are a fine starting point but they drift. As a conversation goes on, the model's training distribution pulls it back toward the helpful-assistant voice, the hidden state block in the prompt gets ignored, and adversarial users can break the persona with a couple of pointed questions. Post-training a small model on a preference dataset that rewards in-character continuations and punishes breaks keeps the persona stable over long runs.

The pipeline lives in ensemble-train, a separate uv workspace member because its dependencies are heavy. It does three things, each in its own module.

spec.py loads a persona TOML into a typed PersonaSpec. If the TOML has a [persona.training] table, spec.is_trainable is true and spec.training holds the LoRA and DPO hyperparameters along with the base model and adapter name.

self_play.py turns the spec into a list of PreferencePairs. Each pair carries a prompt, a chosen continuation that stays in character, and a rejected continuation that breaks character. The default generator is an offline heuristic that builds five prompt and pair templates from the persona's style fields. For real runs, pass a breaker callable that drives a frontier model against the persona and uses a small judge prompt to label which continuations are breaks.

trainer.py runs TRL's DPOTrainer over a LoRA adapter on the configured base model. It saves the adapter to output_dir/ and emits a persona-card.md summary of the run. When HF_USERNAME is set in the environment and the spec names an adapter_name, the adapter is pushed to the Hugging Face hub at completion.

Minimal example

Building the preference dataset needs no GPU and no Torch, so it is the easiest entry point: 

from ensemble_train import generate_preferences, load_persona

spec = load_persona("examples/plank/personas/frustrated_power_user.toml")

The training step needs the Torch extras and a GPU (or a tiny base model for CPU smoke tests). Wire it together end to end: 

from pathlib import Path

from ensemble_train import generate_preferences, load_persona
from ensemble_train.trainer import TrainedAdapter, train

spec = load_persona("examples/plank/personas/frustrated_power_user.toml")
pairs = generate_preferences(spec)
adapter: TrainedAdapter = train(
    spec,
    pairs,
    output_dir=Path("checkpoints/frustrated_power_user"),
)

print(f"trained {adapter.persona_name} on {adapter.base_model}")
print(f"adapter saved to {adapter.output_dir}")
if adapter.pushed_to_hub:
    print(f"pushed to {adapter.pushed_to_hub}")

Backends

Three backends ship out of the box. They share the same train(spec, pairs, output_dir) signature and produce the same TrainedAdapter result, so you can swap between them without touching scenario code.

Local. Runs the trainer in process. CPU works for smoke tests against a tiny base model; GPU works for everything else. This is the backend to reach for when you are iterating on the data pipeline or the training loop.

Modal. Ships the trainer to Modal on an A100, with a one-hour default timeout. The stub installs Torch, Transformers, TRL, PEFT, and Accelerate on a fresh debian_slim image and runs train(...) inside the container. When the modal package is not installed locally, the backend falls back to a dry-run that writes DRY_RUN.json describing what it would have submitted, so the rest of the pipeline still works.

SkyPilot. Renders a SkyPilot YAML next to the persona and calls sky launch -y if sky is on PATH. When it is not, the backend writes the YAML and prints the dispatch command for you to run manually. This is the backend to use when you want spot capacity, or when you are sharing infrastructure with other SkyPilot jobs.

Custom breakers

The offline breaker is deliberately small: five prompt templates that try to drop the persona, paired with in-character continuations sampled from the persona's style fields and rejected continuations that reveal the system prompt. This is enough to exercise the pipeline end to end without API keys.

For real runs you pass a breaker callable that drives a frontier model: 

from ensemble_train import PersonaSpec, PreferencePair, generate_preferences, load_persona

The framework deliberately does not ship a frontend for orchestrating breaker rollouts. That part is moving quickly and the prompt strategy matters more than the wiring; the framework provides the hook and the trainer.

Serving a trained persona

After training, the adapter lives at checkpoints/<persona>/. Two paths take it from there back into a scenario.

The first is the Hugging Face hub. With HF_USERNAME set, the trainer pushes the adapter on completion under the name given in the persona TOML. From there you can pull it onto any vLLM server.

The second is local. Run a vLLM server with the base model and the LoRA adapter loaded, then point Ensemble at it through the OpenAI-compatible HTTP endpoint: 

from ensemble import World

world = World(
    "plank",
    backend="vllm",
    base_url="http://localhost:8000/v1",
)
alice = world.spawn_user(
    id="alice",
    persona="frustrated_power_user",
    model="ensemble-plank-frustrated-power-user",
)

Inside the runtime, LocalAdapterBackend posts to a standard OpenAI Chat Completions endpoint with the adapter name in the model field. vLLM and other OpenAI-compatible servers (LM Studio, OpenRouter, and similar) route requests to the right adapter on that basis.

Architecture

The framework is small on purpose. Three Rust crates and one Python package do all the work.

ensemble-core

The core crate holds the simulation primitives. It has no LLM calls, no HTTP, and no Python. The bus is an mpsc-based router that gives every actor its own inbox and appends every send to a shared event log. The scheduler runs one Tokio task per actor and a watcher that checks the until predicate after every new event. It halts on the predicate, on a tick or event budget, or on quiescence, which is a timeout that fires when no traffic has flowed for a configurable window. Quiescence is what keeps tests from deadlocking when a scripted conversation runs out of turns.

The World type wraps your WorldState behind a Tokio mutex. Its apply_and_log method dispatches a tool call, then emits both a tool_result event with the effect and a state_diff event with the diff. The until combinators (turn_count_exceeds, any_of, all_of) are small labelled closures over the log.

ensemble-runtime

The runtime crate is everything the core deliberately does not touch. The LLMBackend trait has one method, complete, which takes a CompletionRequest and returns a CompletionResponse. Four implementations ship in the crate: an Anthropic Messages API client, an OpenAI Chat Completions client, a vLLM-shaped local-adapter client for serving trained personas, and a deterministic mock backend that backs every test in the repo.

The concrete UserActor and AgentActor live here. Both hold a shared backend and a chat history; the agent also holds a tool registry and dispatches any tool calls the model returns. The PromptedPersona wrapper composes any backend with a system prompt template and a tagged hidden-state block.

Choosing a backend

The backend argument to World(...) and run_scenario(...) takes one of "mock" (the deterministic default), "anthropic", "openai", "vllm", or "auto". "auto" picks the first provider with an API key in the environment, in this order: Anthropic first (the ANTHROPIC_API_KEY variable), then OpenAI (OPENAI_API_KEY), then fallback to mock. When both keys are set, Anthropic wins; pass an explicit backend="openai" if you want the other.

Two small divergences between the shipped backends are worth knowing about. The OpenAI client sends max_completion_tokens (every chat-completions model since mid-2024 accepts it, and newer models like o1 and gpt-5 reject the legacy max_tokens field). The Anthropic client sends max_tokens because that is what the Messages API expects. The vLLM-shaped client follows the OpenAI shape but uses max_tokens, because vLLM accepts both and many of the smaller open models people serve through it haven't kept up with the newer name. The single max_tokens field on CompletionRequest is translated to whichever name each provider wants.

Each backend, on completion, parses its provider's usage block (Anthropic's input_tokens and output_tokens, OpenAI's prompt_tokens and completion_tokens) and records tokens_in, tokens_out, and (when the model is in crates/ensemble-runtime/pricing.toml) usd against the calling actor. Pair this with per-actor budgets to cap a slot's spend without affecting the rest of the run.

ensemble-py and the bindings

python/ensemble/ is a maturin-built mixed Rust and Python package. The Rust side exposes a small surface of pyo3 classes. The Python side wraps those with the friendlier @scenario, World, User, and Agent types you see in scenario code. Everything async runs on a process-global Tokio runtime. world.run(until) blocks the calling thread through block_on. async with world.simulate() spawns the scheduler in the background and exposes wait_until via asyncio.to_thread, which keeps the asyncio event loop responsive while the native scheduler runs.

ensemble-train

The training package is a separate uv workspace member because its dependencies are heavy. The full walkthrough lives in the Training personas section; in short, spec.py loads a persona TOML, self_play.py produces preference pairs, and trainer.py wraps TRL's DPOTrainer over a LoRA adapter. Three backends ship out of the box: local, modal, and skypilot.

The CLI

crates/ensemble-cli is a Clap-based binary. It does not contain any business logic of its own. The run command shells out to uv run python with a small script that imports the scenarios package and dispatches through the registry. The train command shells out to uv run ensemble-train. The trace view command is a static HTTP server that serves the site/ directory with the supplied trace dropped in as trace.jsonl.

Trace format reference

Reference: traces.

Traces are JSONL files. One line per event, ordered by tick: 

{
  "tick": 7,
  "ts_ms": 1716000123,
  "actor": "rep1",
  "message_id": "msg-...",
  "payload": {
    "kind": "tool_call",
    "name": "issue_refund",
    "args": { "user_id": "u-alice", "amount_cents": 15000, "reason": "..." }
  }
}

Every event has these top-level fields. The payload kind distinguishes the body:

kindemitted bypayload fields
user_messageUserActortext
agent_messageAgentActortext
tool_callAgentActor or scenario seedname, args
tool_resultToolRegistry dispatchname, result
state_diffWorldState::apply via apply_and_logdiff
systemscheduler stop reasons, runtime notesnote

The trace is intended to be human-readable. When you are debugging a scenario, cat traces/foo.jsonl | jq is often faster than opening the viewer.

Contributing

The fastest way to get involved is to add a new persona or a new scenario to Plank. Both are TOML or Python files, and both show up immediately in ensemble run and the test suite.

Adding a new world is the next step up. Run ensemble init my_world for a working skeleton, implement the WorldState trait, register your tools with a fresh ToolRegistry, and write a smoke scenario. You will know it is working when cargo test and uv run pytest tests/ both stay green.

A few house rules keep the codebase consistent.

  • Tests over screenshots. Both test suites stay green on every commit.
  • No em dashes anywhere in code, docs, or commit messages.
  • No banner-style separator comments. Section comments are fine if they are a single line of normal prose.
  • Commit messages sound like a person wrote them. Active voice, present tense, no Conventional Commits prefixes.
  • One logical change per commit. Split big changes rather than bundling them.
