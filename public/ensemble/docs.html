Quickstart

You need Rust 1.80+, Python 3.10+, and uv. Clone the repo, then: 

uv sync                              # build the python extension, install workspace
cargo build -p ensemble-cli           # produces ./target/debug/ensemble
cargo test --workspace                # 8 rust tests
uv run pytest tests/                  # 21 python tests

Run the hero demo against the mock backend:

uv run python examples/plank/bake_trace.py
./target/debug/ensemble trace view site/trace.jsonl --site site --port 8765
# open http://localhost:8765

Run a scenario for real against a registered LLM:

export ANTHROPIC_API_KEY=...
./target/debug/ensemble run plank.refund_storm --world plank
# trace lands at ./traces/plank_refund_storm.jsonl

Hello world, the smallest scenario you can write:

from ensemble import scenario

@scenario("hello")
async def hello(world):
    alice = world.spawn_user(id="alice", model="user-model")
    rep   = world.spawn_agent(id="rep",  model="agent-model")
    alice.say("rep", "hi")
    yield world.until(world.turn_count > 4)
    yield {"ok": 1.0}

The @scenario decorator registers your function in a global registry. run_scenario("hello") drives it end to end and returns a RunResult with the trace and the grader scores.

Mental model

Three concepts do most of the work.

The world is the typed, mutable state of the simulation. You implement WorldState in Rust. Plank's world is SQLite with users, tickets, subscriptions, knowledge-base articles, and an audit log.

Actors are tokio tasks with mpsc inboxes. A user actor and an agent actor both wrap an LLMBackend. Users emit messages; agents emit messages and tool calls. Each tool call dispatches through a ToolRegistry and produces a ToolResult event on the shared bus.

Scenarios compose actors, seed initial state, run the simulation until an until condition fires, then compute graders. The same simulation can be driven by a Python function with two yields, an async with block for mid-run intervention, or a TOML manifest with a safe boolean grader DSL.

ensemble-core

Pure simulation primitives. No LLM calls, no HTTP, no Python. Five small modules do all the work:

  • bus.rs, the message router. Holds per-actor mpsc inboxes and a shared EventLog. Every bus.send writes a typed event to the log and wakes the scheduler watcher via a Notify.
  • scheduler.rs, one tokio task per actor draining its inbox plus a watcher that checks the until predicate after every new event. Halts on the predicate, on a tick or event budget, or on quiescence (no traffic for quiescence_ms). The quiescence path is what keeps tests from deadlocking when a scripted conversation runs out of turns.
  • world.rs, the WorldState trait and a WorldHandle<S> that wraps it behind a tokio mutex. apply_and_log dispatches a tool call and emits both ToolResult and StateDiff events.
  • event.rs, the typed event payload, the EventLog, and JSONL serialization. Events carry tick, timestamp, actor id, message id, and a typed payload (UserMessage, AgentMessage, ToolCall, ToolResult, StateDiff, System).
  • until.rs, the halting-condition combinators. Until is a labelled closure over UntilCtx. Built-ins: turn_count_exceeds, any_of, all_of.

The WorldState trait is the contract a world author implements: 

pub trait WorldState: Send + Sync + 'static {
    type ToolCall:   DeserializeOwned + Send;
    type ToolEffect: Serialize + Send;
    type Diff:       Serialize + Send;

    fn apply(&mut self, call: Self::ToolCall)
        -> Result<(Self::ToolEffect, Self::Diff), ToolError>;

    fn snapshot(&self) -> Vec<u8>;
    fn restore(&mut self, snapshot: &[u8]) -> Result<(), RestoreError>;
}

Diff is what the trace viewer renders on the right side. Make it rich enough that a reader of the JSONL can reconstruct exactly what changed.

ensemble-runtime

Everything the core deliberately does not touch.

  • backend.rs, the LLMBackend trait, the CompletionRequest/Response shapes, and the SharedBackend = Arc<dyn LLMBackend> alias the actors hold.
  • backends/anthropic.rs, the Anthropic Messages API client. Reads ANTHROPIC_API_KEY from the env. Tool use via the standard tools and tool_use content blocks. Non-streaming.
  • backends/openai.rs, Chat Completions with function calling.
  • backends/vllm.rs, the LocalAdapterBackend. Hits a vLLM-compatible OpenAI-shaped endpoint. Pass an adapter name when one base server serves many trained adapters.
  • backends/mock.rs, the deterministic MockBackend. Takes a MockScript of canned turns keyed by model name. This is what backs every test in the repo, and what bakes the trace on the front page.
  • actors.rs, the concrete UserActor and AgentActor. Both hold a SharedBackend and a chat history. The agent also holds an Arc<ToolRegistry> and dispatches any tool calls the model returns.
  • tools.rs, the Tool + ToolRegistry pair. A tool is a name, a description, a JSON Schema, and a closure that maps parsed args to a JSON effect.
  • persona.rs, PromptedPersona wraps any LLMBackend with a system prompt template and a tagged <hidden_state> block. Non-revelation is a training problem; the wrapper just renders the state.

Python bindings

The ensemble Python package is a maturin-built mixed Rust+Python project. The Rust side lives in python/ensemble/src/lib.rs and exposes three pyo3 classes (World, User, Agent) plus a process-global tokio runtime.

The Python side (python/ensemble/scenario.py) wraps those classes with a friendlier surface: the @scenario decorator, an Until dataclass, a TurnCount sentinel that makes world.turn_count > 30 evaluate to an Until, and an async with world.simulate() context manager.

The native bridge is intentionally small. Scenarios call world.run(until), which serializes the until spec to JSON, hands it to Rust, and blocks on the global tokio runtime via block_on. For the async with path the bridge starts the scheduler in the background and provides wait_for_until, which the Python side wraps in asyncio.to_thread so the asyncio loop stays responsive.

We do not depend on pyo3-asyncio's python-awaitable bridge. That keeps the call-site small at the cost of one blocking call per wait_until; the trade-off is documented in CHOICES.md.

Scenarios

Three ways to write the same simulation.

Yield flavor

The common case. Spawn actors, yield an until, yield the grader dict. The first yield kicks off the scheduler; the second yield returns the scores. 

@scenario("refund_storm")
async def refund_storm(world):
    alice = world.spawn_user(persona="frustrated_power_user", hidden_goal="refund_3mo")
    bob   = world.spawn_user(persona="confused_new_user")
    rep1  = world.spawn_agent(model="claude-sonnet-4-5",
                               tools=["lookup_user", "issue_refund", "escalate"])
    alice.act("open_ticket", subject="want my money back")
    bob.act("open_ticket",   subject="how do refunds work")

    yield world.until(world.turn_count > 30)

    yield {
        "alice_refund_resolved": 1.0 if alice.hidden_goal_resolved() else 0.0,
        "no_double_refunds":     float(not world.had_double_refund()),
    }

Async with flavor

Use this when you need to tweak state mid-run, inject a message, or assert an intermediate invariant. The scheduler runs on the global tokio runtime; the context manager starts and stops it. 

@scenario("enterprise_audit")
async def enterprise_audit(world):
    carol = world.spawn_user(persona="enterprise_admin", hidden_goal="audit_log_export")
    rep   = world.spawn_agent(model="claude-sonnet-4-5", tools=["escalate", "search_kb"])
    carol.say("rep", "i need the q3 audit log exported as csv.")

    async with world.simulate() as run:
        await run.wait_until(world.turn_count > 10)
        carol.say("rep", "if you cannot do this, please escalate.")
        await run.wait_until(world.turn_count > 18)

    return {"escalated_or_promised": 1.0}

TOML flavor

Declarative scenarios live in scenarios.toml and compile to the same runtime at load time. Useful when you want to enumerate a matrix of similar scenarios without writing one Python function each. 

[scenario.refund_storm]
world = "plank"
duration_turns = 30

[[scenario.refund_storm.users]]
id = "alice"
persona = "frustrated_power_user"
initial_action = { tool = "open_ticket",
                   args = { subject = "want my money back" } }

[[scenario.refund_storm.agents]]
id = "rep1"
model = "claude-sonnet-4-5"
tools = ["lookup_user", "issue_refund"]

[scenario.refund_storm.graders]
alice_resolved   = "hidden_goal_resolved"
no_double_refund = "not had_double_refund"

Grader expressions parse with a hand-rolled recursive descent parser. Supported tokens are and, or, not, parens, and named predicates the world exposes. No eval, no ast.literal_eval, no calls or attribute access.

Plank

Plank is a small fake SaaS customer-support tool that doubles as the worked example for Ensemble. State is in-memory SQLite with six tables; six tools operate on them; five personas exercise the agents from different angles. 

users         (id, name, email, plan)
tickets       (id, user_id, subject, status, opened_at)
subscriptions (user_id, plan)
kb            (id, title, body)
audit_log     (id, ts, who, action, payload)
refunds       (id, user_id, amount_cents, reason, ts)

The interesting policy is in issue_refund: it counts a user's prior refunds and rejects a second one. This is what lets the no_double_refunds grader actually measure something.

See the plank page for the full tool list and the three shipped scenarios.

Training pipeline

ensemble-train is a separate uv workspace member because its torch/transformers stack is heavy. Three layers:

  • spec.py loads a persona TOML into a typed PersonaSpec with optional TrainingHyperparams.
  • self_play.py turns a persona spec into a list of PreferencePairs. The default is a deterministic offline heuristic; pass a custom breaker callable to plug in a frontier model that tries to push the persona out of character.
  • trainer.py runs TRL's DPOTrainer on a LoRA adapter over the base model, saves to output_dir, optionally pushes to Hugging Face under $HF_USERNAME, and emits a persona-card.md.

Backends live in train/ensemble_train/backends/:

  • local.py, runs in-process; CPU smoke tests against a tiny base model.
  • modal.py, ships the trainer to Modal on an A100; falls back to a dry-run when modal isn't installed.
  • skypilot.py, renders a SkyPilot YAML and launches via sky launch; dry-runs (writes the YAML) when sky isn't on PATH.

Kick off a real training run:

uv pip install 'ensemble-train[torch]'
./target/debug/ensemble train examples/plank/personas/frustrated_power_user.toml \
  --backend modal

CLI

The ensemble binary is a thin wrapper around the python and rust libraries.

ensemble init <name>                  scaffold a new world project
ensemble run <scenario> [--world W]   run, write trace to ./traces/
ensemble trace view <path>            serve the trace viewer with the file baked in
ensemble train <persona.toml>         hand off to ensemble-train

run shells out to uv run python -c <script> that imports the scenarios package, calls the scenario via the registry, and writes the trace. train shells out to uv run ensemble-train. trace view serves site/ on http://localhost:<port> with the given trace dropped in at trace.jsonl.

Trace format

JSONL, one line per event, ordered by tick.

{
  "tick": 7,
  "ts_ms": 1716000123,
  "actor": "rep1",
  "message_id": "msg-...",
  "payload": {
    "kind": "tool_call",
    "name": "issue_refund",
    "args": {"user_id": "u-alice", "amount_cents": 15000, "reason": "..."}
  }
}

Event kinds:

kindemitted byfields
user_messageUserActortext
agent_messageAgentActortext
tool_callAgentActor or scenario seedname, args
tool_resultToolRegistry dispatchname, result
state_diffWorldState::apply via apply_and_logdiff
systemscheduler stop reasons, runtime notesnote

Contributing

The fastest way to get involved is to add a new persona or a new scenario to Plank. Both are TOML files; both immediately show up in ensemble run and in the test suite.

Adding a new world is the next step up. Run ensemble init my_world to get a Rust crate skeleton with a WorldState stub, a Python scenario, a TOML manifest, and a persona TOML. Implement the trait, register your tools, and you are off.

House rules:

  • Tests over screenshots. cargo test --workspace and uv run pytest tests/ both stay green.
  • No em-dashes, no banner-style separators in comments.
  • Commits sound like a person wrote them: present tense, no Conventional Commits prefixes.
  • One logical change per commit. Split big changes; don't bundle.
