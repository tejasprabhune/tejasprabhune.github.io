MCP

Ensemble exposes a scenario's tools over the Model Context Protocol so external MCP-aware clients (Claude Code, Codex, Claude Desktop) can take over one named agent slot inside a running scenario. The trace records the slot's activity exactly as it would record a local agent's.

The shape of an MCP-driven slot

A scenario spawns agents with world.spawn_agent; normally each agent has a backend that decides what to say and which tool to call. ensemble mcp serve replaces one of those agents with an external proxy: the scheduler keeps routing messages to its inbox, but the responses come from a connected MCP client instead of an LLM the scenario configured. The client sees three kinds of tools:

  • The world's tools (filtered to the slot's allowed list).
  • inbox_recv, a meta tool that returns the next message routed to the slot (or {empty: true}).
  • agent_say, a meta tool that sends a message from the slot to another actor.

Tool calls dispatched by the client land in the trace with the slot's id as the acting actor, so a grader written against a normal rollout still works against an MCP-driven rollout.

Running mcp serve

Run the CLI from the repo root after the plank world is registered: 

ensemble worlds add plank examples/plank
ensemble mcp serve --world plank --scenario plank.refund_storm --as-agent rep1

The process serves MCP on stdio. The scenario runs in a background thread; when the until predicate fires the scheduler halts and the server exits. With --backend mock (the default) the other actors use scripted responses; pass --backend anthropic to give the non-MCP actors a real model.

Driving the slot

Any MCP client that speaks stdio can drive the slot. The snippet below uses the official Python MCP SDK directly; in practice you would point Claude Code, Claude Desktop, or Codex at the same command. 

# client.py — drive a slot through ensemble's MCP server.
import asyncio
from mcp import ClientSession
from mcp.client.stdio import StdioServerParameters, stdio_client

params = StdioServerParameters(
    command="ensemble",
    args=["mcp", "serve",
          "--world", "plank",
          "--scenario", "plank.refund_storm",
          "--as-agent", "rep1"],
)

async def main():
    async with stdio_client(params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()
            tools = await session.list_tools()
            print([t.name for t in tools.tools])
            msg = await session.call_tool("inbox_recv", arguments={})
            print(msg.content[0].text)

asyncio.run(main())

Caveats

  • The server uses the official Python MCP SDK (mcp.server.lowlevel.Server). The rust rmcp crate is mature enough to host the server itself; ensemble chose the Python path because plugin tools are already Python callables and an rmcp host would re-acquire the GIL on every dispatch. See the inline note at python/ensemble/mcp_server.py for the upgrade path.
  • Only stdio is supported today. SSE / streamable HTTP land in a follow-up; the protocol shape is identical.
  • Per-tool timeouts and resource locks still apply to MCP-dispatched calls (they go through the same ResourceManager the agent loop uses).
